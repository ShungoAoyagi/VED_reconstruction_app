API仕様の前後でフロント側が詰めておくと実装が止まりにくい論点は次あたりです。
ゲーム進行の状態遷移
Start -> LevelSelect -> Playing -> Result -> Ranking の遷移条件を厳密化（中断時・時間切れ・不正解で終了時・正解即終了時）。
とくに「最後の解答時刻を採用」「不正解時は最良スコア採用」の判定タイミングを状態機械として定義しておくと安全です。
タイマー仕様の境界条件
「チュートリアルモーダル中もタイマー停止しない」があるので、setInterval ではなく単調時刻ベース（開始時刻との差分）で管理する方針を決める。
背景化・スリープ復帰時のズレ扱いも決めるべきです。
採点計算の数値仕様
規格化方法、内積の離散化（グリッド解像度）、丸め（切り上げ）と浮動小数誤差の扱いを先に固定。
フロント表示値とバックエンド採点値が1点でもズレるとUXが悪いので、表示用・公式値の責務分離が必要です。
乱数と再現性
「位相の原点ランダム」「軌道セット選択」があるため、1プレイ内での固定単位（開始時に確定して固定か、回答ごと再抽選か）を明文化。
デバッグ/検証用に seed を持つかも決めておくと後で楽です。
入力制約とバリデーションUX
難易度ごとに係数・位相制約が異なるので、Input/SelectButton の制約（小数桁、範囲外、空値、IME入力途中）をUI仕様として先に確定。
エラー表示位置（position: static 以外）も、レイアウト崩れ対策ルールまで決めると実装しやすいです。
3D表示コンポーネントの責務分割
回転同期・動的更新・複数ビュー同期があるため、
「描画エンジン層」「共有カメラ状態層」「フォーム入力層」を分ける設計を先に決めるべきです。
ここを曖昧にすると再レンダリングで重くなりやすいです。
ローカルランキングの永続化仕様
top5更新条件、同点順位、同名扱い、保存先（Electronならapp.getPath('userData')配下想定）、破損時復旧を先に定義。
“ローカル” なので複数OSでのファイルパス差も仕様化しておくと安心です。
Electron特有の境界
preload 経由で何を公開するか（最小権限）、FastAPI起動失敗時のUI（リトライ/再起動導線）を要件化。
「localhost:8000固定か」「ポート競合時の挙動」も先に決めるべきです。
実装前ドキュメントとしては、次の3点を追加すると強いです。
画面ごとの状態遷移図（イベント/ガード条件付き）
各難易度の入力・採点・終了条件の真理値表
3D表示とフォームのデータフロー図（どこが single source of truth か）
必要なら次に、AGENT.md に追記するための「そのまま貼れる要件テンプレート（状態遷移・バリデーション・エラー時挙動）」を作ります。
